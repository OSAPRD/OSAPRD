# Upload Curation Data Pipeline

This package turns local curation output into a public Hugging Face dataset
layout and optionally uploads it. It does not run extraction, curation,
topic-classification training, analysis, or metric computation.

The stage has one input source and two commands:

- `prepare`: read local curation output and write public entity parquet plus
  metadata locally.
- `upload`: upload an already prepared local staging directory to a Hugging Face
  dataset repository.

Use `all` to run `prepare` and `upload` in sequence.

## Pipeline Flow

Upload-curation has three phases:

```text
CLI/env settings
  -> prepare: stream local curation aggregates into public entity tables
  -> manifest: write schema, dataset card, run manifest, and optional upload plan
  -> upload: publish the staged tree to a Hugging Face dataset repository
```

Prepare reads curation aggregate JSON, snapshot manifests, repository file
lists, optional topic-classification outputs, and optional longitudinal
refactoring summaries. It writes one public dataset layout with cohort-scoped
tables under `data/<cohort>/Curation/` and repository-level tables under
`data/Curation/`.

The maintainability upload table contains the current metric set:

- `maintainability_index`
- `cyclomatic_complexity`
- `halstead_volume`
- `fan_out`
- `comment_ratio`
- `loc`
- `duplicated_lines_density`

The first six metrics come from Multimetric output. `duplicated_lines_density`
is the custom deterministic duplication metric generated by curation and
packaged with the Multimetric rows.

## Packaged Analysis Components

The prepared dataset keeps the analysis components separate so downstream users
can join only the pieces they need:

- Curated PR metadata: retained PRs, cohort labels, sampler metadata, commit
  identifiers, repository identifiers, and snapshot availability.
- Refactoring results: original PR refactoring summaries from curation plus
  optional longitudinal refactoring summaries when a longitudinal output root is
  supplied.
- Maintainability results: Multimetric maintainability values and custom
  duplicated-lines density for each curation snapshot.
- Code-smell results: curation code-smell count summaries by tool, language,
  snapshot, and PR.
- Snapshot manifests: before, after, and future snapshot metadata plus
  de-duplicated source file references.
- Repository metadata: repository-level metadata, file-list references, and
  optional topic-classification labels.

The uploader does not invoke analysis tools. It packages outputs produced by
curation, including refactoring results from
[RefactoringMiner](https://github.com/tsantalis/RefactoringMiner),
[RefDiff](https://github.com/aserg-ufmg/RefDiff), and
[RefactoringMiner++](https://github.com/benzoinoo/RefactoringMinerPP);
maintainability results from
[Multimetric](https://github.com/priv-kweihmann/multimetric) plus custom
duplicated-lines density; and code-smell summaries from
[DesigniteJava](https://www.designite-tools.com/products-dj),
[DesignitePython/DPy](https://www.designite-tools.com/products-dpy),
[PMD](https://pmd.github.io/), [ESLint](https://eslint.org/),
[Cppcheck](https://cppcheck.sourceforge.io/), and
[clang-tidy](https://clang.llvm.org/extra/clang-tidy/).

## Hugging Face APIs Used

This stage uses the [Hugging Face Hub Python API](https://huggingface.co/docs/huggingface_hub/en/package_reference/hf_api)
through `huggingface_hub.HfApi`.

Uploads use [`upload_large_folder`](https://huggingface.co/docs/huggingface_hub/en/guides/upload#upload-a-large-folder)
for parquet data because curation packages can contain many shards and snapshot
blob files. Metadata files are uploaded with `upload_file`.

Authentication uses Hugging Face environment variables. The preferred variable
is `HF_TOKEN`; `HUGGINGFACE_HUB_TOKEN` and `HUGGING_FACE_HUB_TOKEN` are also
accepted. Hugging Face documents `HF_TOKEN` and `HF_HOME` in its
[environment variable reference](https://huggingface.co/docs/huggingface_hub/en/package_reference/environment_variables).

## Rate Limits and Upload Resumability

Large dataset uploads can hit Hugging Face request or repository commit quotas.
This stage handles those limits by retrying transient failures, pausing for
short-term and hourly quota messages, and using the same local output directory
as the upload cache boundary.

Resumability is local:

- `upload_curation_state.sqlite3` records uploaded repo paths and snapshot blob
  de-duplication state.
- Re-running `prepare` with the same output directory continues parquet batch
  numbering from existing staged files.
- Re-running `upload` with the same output directory reuses Hugging Face upload
  cache files and the SQLite upload state.
- Use a fresh output directory for an independent staged package.

## Token Handling

Set Hugging Face tokens only through environment variables:

- `HF_TOKEN`: preferred.
- `HUGGINGFACE_HUB_TOKEN`: fallback.
- `HUGGING_FACE_HUB_TOKEN`: fallback.

The token is used only for real uploads. `prepare` and `upload --dry-run` do not
require a token. Manifests record only whether a token was configured and the
environment-variable lookup order; they never write the token value.

PowerShell:

```powershell
$env:HF_TOKEN = "hf_..."
```

Bash:

```bash
export HF_TOKEN=hf_...
```

## Input Sources and Configuration

The required input is a local curation output root. The uploader discovers
cohort directories and aggregate files under that root; it does not compute new
metrics or run external tools.

Expected curation input shape:

```text
/absolute/path/to/curation-output/
  humans/
    sampled_prs_*.jsonl
    longitudinal_prs_*.jsonl
    output/
      processed-data/
        <run-or-repository>/
          metrics-json/
            *.json
      snapshots/
        <owner>/
          <repo>/
            repository_file_list.json
    ...
  codex/
    sampled_prs_*.jsonl
    output/
      processed-data/
        <run-or-repository>/
          metrics-json/
            *.json
    ...
  claude/
    ...
```

Optional side inputs:

- Topic-classification output root with `topic_classification_manifest.json`
  and repository topic result files.
- Longitudinal-refactoring output root with
  `longitudinal_refactoring_manifest.json`,
  `longitudinal_refactoring_snapshot_results.jsonl`, and optional operation or
  PR-summary JSONL files.

Missing optional side inputs do not block packaging. Use empty directories in
Docker when those outputs are not available.

Configuration follows the same precedence as the extraction and curation
stages:

```text
CLI flags > POST_PROCESSING_UPLOAD_CURATION_* environment variables > defaults
```

The active configuration modules are:

- `config/settings.py`: resolves one typed `UploadCurationSettings` object and
  redacts secrets before writing manifests.
- `config/storage_config.py`: path, batching, compression, schema, and retry
  defaults.
- `config/tokens_config.py`: Hugging Face token lookup from environment
  variables.

Useful environment fallbacks:

- `POST_PROCESSING_CURATION_OUTPUTS_DIR`: fallback for the curation input root.
- `POST_PROCESSING_TOPIC_CLASSIFICATION_OUTPUTS_DIR`: fallback for the topic
  side-input root.
- `POST_PROCESSING_LONGITUDINAL_REFACTORING_OUTPUTS_DIR`: fallback for the
  longitudinal side-input root.
- `HF_DATASET_REPO_ID`: fallback for the Hugging Face dataset repository id.

## Requirements

Docker is the recommended way to run upload-curation because it isolates the
Python parquet stack and Hugging Face client from the host environment. The
image installs `post-processing/upload-curation-data/requirements.txt`.

For Docker runs, you need:

- Docker.
- Local curation output mounted read-only to `/data/input`.
- Optional topic-classification output mounted read-only to
  `/data/topic-classification`.
- Optional longitudinal-refactoring output mounted read-only to
  `/data/longitudinal-refactoring`.
- A local staging/output directory mounted writable to `/data/output`.
- A Hugging Face dataset repository id when uploading.
- `HF_TOKEN` with write access when uploading.

For local runs without Docker, use Python 3.12. The dependency file is
stage-scoped:

- `huggingface_hub`: repository creation and upload.
- `pandas`, `pyarrow`, and `numpy`: parquet reading/writing and schema handling.

## Step-by-Step Run: Docker

Use this path first unless you are actively debugging the Python process on the
host.

1. Prepare local paths.

   Use a curation output root with aggregate output and cohort directories, for
   example:

   ```text
   /absolute/path/to/curation-output/
     humans/
     codex/
     claude/
     ...
   ```

   Topic-classification and longitudinal-refactoring outputs are optional. If
   they are unavailable, mount an empty directory.

   Create a writable staging directory:

   ```bash
   mkdir -p outputs/upload-curation
   mkdir -p outputs/empty-topic-classification
   mkdir -p outputs/empty-longitudinal-refactoring
   ```

2. Build the image from the repository root.

   ```bash
   docker build \
     -f post-processing/upload-curation-data/Dockerfile \
     -t mosaic-upload-curation:local \
     .
   ```

3. Check the container CLI.

   ```bash
   docker run --rm mosaic-upload-curation:local \
     python post-processing/upload-curation-data/run.py --help
   ```

4. Prepare a local public parquet package without uploading.

   ```bash
   docker run --rm \
     -v /absolute/path/to/curation-output:/data/input:ro \
     -v "$(pwd)/outputs/empty-topic-classification:/data/topic-classification:ro" \
     -v "$(pwd)/outputs/empty-longitudinal-refactoring:/data/longitudinal-refactoring:ro" \
     -v "$(pwd)/outputs/upload-curation:/data/output" \
     mosaic-upload-curation:local \
     python post-processing/upload-curation-data/run.py prepare
   ```

5. Inspect the prepared output.

   Check `outputs/upload-curation/data/<cohort>/Curation/` for entity parquet
   batches, `outputs/upload-curation/schema_manifest.json` for table
   definitions, and `outputs/upload-curation/upload_curation_manifest.json` for
   run metadata.

6. Create a dry-run upload plan.

   ```bash
   docker run --rm \
     -v "$(pwd)/outputs/upload-curation:/data/output" \
     -e POST_PROCESSING_UPLOAD_CURATION_REPO_ID="your-org/your-dataset" \
     mosaic-upload-curation:local \
     python post-processing/upload-curation-data/run.py upload --dry-run
   ```

   Inspect `outputs/upload-curation/upload_curation_plan.json` before a real
   upload.

7. Upload the prepared tree.

   ```bash
   export HF_TOKEN=hf_...

   docker run --rm \
     -e HF_TOKEN="$HF_TOKEN" \
     -e POST_PROCESSING_UPLOAD_CURATION_REPO_ID="your-org/your-dataset" \
     -v "$(pwd)/outputs/upload-curation:/data/output" \
     mosaic-upload-curation:local \
     python post-processing/upload-curation-data/run.py upload
   ```

8. Run prepare and upload in one command when the source and destination are
   already confirmed.

   ```bash
   docker run --rm \
     -e HF_TOKEN="$HF_TOKEN" \
     -e POST_PROCESSING_UPLOAD_CURATION_REPO_ID="your-org/your-dataset" \
     -v /absolute/path/to/curation-output:/data/input:ro \
     -v "$(pwd)/outputs/empty-topic-classification:/data/topic-classification:ro" \
     -v "$(pwd)/outputs/empty-longitudinal-refactoring:/data/longitudinal-refactoring:ro" \
     -v "$(pwd)/outputs/upload-curation:/data/output" \
     mosaic-upload-curation:local \
     python post-processing/upload-curation-data/run.py all
   ```

## Step-by-Step Run: Local

Use the local path when Docker is unavailable.

1. Install dependencies from the repository root.

   ```bash
   python -m pip install -r post-processing/upload-curation-data/requirements.txt
   ```

2. Check the CLI.

   ```bash
   python post-processing/upload-curation-data/run.py --help
   ```

3. Prepare local public parquet output.

   ```bash
   python post-processing/upload-curation-data/run.py prepare \
     --input-dir /absolute/path/to/curation-output \
     --topic-classification-dir /absolute/path/to/topic-classification-output \
     --longitudinal-refactoring-dir /absolute/path/to/longitudinal-refactoring-output \
     --output-dir outputs/upload-curation
   ```

4. Dry-run the upload.

   ```bash
   python post-processing/upload-curation-data/run.py upload \
     --output-dir outputs/upload-curation \
     --repo-id your-org/your-dataset \
     --dry-run
   ```

5. Upload.

   ```bash
   export HF_TOKEN=hf_...

   python post-processing/upload-curation-data/run.py upload \
     --output-dir outputs/upload-curation \
     --repo-id your-org/your-dataset
   ```

Important CLI options:

- `prepare|upload|all`: command to run.
- `--input-dir`: curation output root.
- `--exclude-dir`: source-run directory name to skip. Repeat as needed.
- `--topic-classification-dir`: optional topic-classification output root.
- `--topic-classification-top-k-topics`: maximum public topic labels per repo.
- `--longitudinal-refactoring-dir`: optional longitudinal-refactoring output
  root.
- `--output-dir`: local staging directory.
- `--repo-id`: Hugging Face dataset repository id.
- `--dry-run`: write an upload plan without contacting Hugging Face.
- `--data-subdir`: staged data subdirectory, default `data`.
- `--output-batch-size`: rows per parquet shard.
- `--max-files-per-directory`: shard directories before this file count.
- `--parquet-compression`: parquet compression codec.
- `--state-db-filename`: SQLite state filename under the output directory.
- `--schema-version`: public curation schema version recorded in manifests.
- `--blob-batch-bytes`: approximate snapshot-file blob parquet batch threshold.

Environment equivalents:

- `POST_PROCESSING_UPLOAD_CURATION_INPUT_DIR`
- `POST_PROCESSING_UPLOAD_CURATION_TOPIC_CLASSIFICATION_DIR`
- `POST_PROCESSING_UPLOAD_CURATION_LONGITUDINAL_REFACTORING_DIR`
- `POST_PROCESSING_UPLOAD_CURATION_OUTPUT_DIR`

## Output Layout

Prepared output:

```text
upload-curation-output/
  README.md
  schema_manifest.json
  upload_curation_manifest.json
  upload_curation_plan.json
  upload_curation_state.sqlite3
  data/
    Curation/
      RepositoryMetadata/
      RepositoryTopicClassifications/
    codex/
      Curation/
        CuratedPullRequests/
        RefactoringResults/
        MaintainabilityResults/
        CodeSmellResults/
        SnapshotManifests/
        SnapshotFileRefs/
        SnapshotFileBlobs/
    humans/
      Curation/
        ...
```

Entity tables:

- `CuratedPullRequests`: one PR-level row per retained curation aggregate.
- `RefactoringResults`: original PR refactoring summaries and optional
  longitudinal refactoring summaries.
- `MaintainabilityResults`: Multimetric maintainability values plus custom
  duplicated-lines density per snapshot.
- `CodeSmellResults`: code-smell count summaries by snapshot.
- `RepositoryMetadata`: repository-level metadata and repository file-list
  references.
- `RepositoryTopicClassifications`: optional topic-classification labels.
- `SnapshotManifests`: one row per before, after, or future snapshot.
- `SnapshotFileRefs`: snapshot file references that point at de-duplicated blob
  rows.
- `SnapshotFileBlobs`: de-duplicated snapshot file contents.

Metadata files:

- `README.md`: generated dataset card for the staged package.
- `schema_manifest.json`: entity schemas, primary keys, and foreign keys.
- `upload_curation_manifest.json`: safe settings, counts, schema version,
  metric list, and upload-plan summary.
- `upload_curation_plan.json`: dry-run upload file list and target repo paths.
- `upload_curation_state.sqlite3`: local upload/blob de-duplication state.

## Resume

Use the same `--output-dir` to resume a staged package.

Prepare resume behavior:

- Existing parquet output batch numbers are detected so new batches continue at
  the next index.
- Snapshot file blobs are de-duplicated by SHA-256 content hash.

Upload resume behavior:

- `upload_large_folder` reuses its local cache under the staged output tree.
- Uploaded repo paths are recorded in SQLite.
- If an upload is interrupted, rerun `upload` with the same `--output-dir`.

Use a fresh output directory for a completely independent package.

## Troubleshooting

No curation aggregate files are discovered:

- Confirm `--input-dir` points at the curation output root, not the upload
  staging directory.
- Confirm source-run directories were not excluded through `--exclude-dir` or
  `POST_PROCESSING_UPLOAD_CURATION_EXCLUDE_DIRS`.
- In Docker, confirm the host path is mounted to `/data/input`.

Topic rows are missing:

- Confirm `--topic-classification-dir` points at a completed
  topic-classification output root.
- Topic-classification output is optional; missing topic rows do not block
  curation packaging.

Longitudinal refactoring rows are missing:

- Confirm `--longitudinal-refactoring-dir` points at a directory containing
  `longitudinal_refactoring_manifest.json` and JSONL result files.
- Longitudinal output is optional; missing longitudinal rows do not block
  curation packaging.

Upload is skipped:

- Confirm `--repo-id` or `POST_PROCESSING_UPLOAD_CURATION_REPO_ID` is set.
- Confirm `HF_TOKEN` is set for a real upload.
- `upload --dry-run` never contacts Hugging Face.

Upload hits rate limits:

- Rerun the same command after the cooldown window.
- Keep the same output directory so upload cache and SQLite state are reused.
- Reduce `--upload-large-folder-num-workers` if request pressure is too high.
- Increase `--upload-large-folder-directory-cooldown-seconds` for large exports.

## Files of Interest

- `post-processing/upload-curation-data/run.py`: CLI entrypoint and command
  dispatcher.
- `post-processing/upload-curation-data/config/settings.py`: typed settings
  with CLI > environment > defaults precedence.
- `post-processing/upload-curation-data/config/storage_config.py`: input,
  output, batching, compression, and retry defaults.
- `post-processing/upload-curation-data/config/tokens_config.py`: Hugging Face
  token loading and redacted token metadata.
- `post-processing/upload-curation-data/hf_curation_uploader.py`: streaming
  conversion, local manifests, SQLite state, and upload orchestration.
- `post-processing/upload-curation-data/Dockerfile`: Docker runtime.
- `post-processing/upload-curation-data/Dockerfile.dockerignore`: Docker
  context exclusions for local data and tool caches.
- `post-processing/upload-curation-data/requirements.txt`: stage-scoped Python
  dependencies.

The Python files use module docstrings for stage contracts and short comments
for non-obvious decisions such as schema stability, token redaction, retry
policy, and resumable state. Routine line-by-line behavior is intentionally left
to the code.
